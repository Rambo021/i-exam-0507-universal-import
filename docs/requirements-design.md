# 万能导入 —— 多模板自动导入下单系统需求与设计文档

## 1. 需求理解

本项目需要实现一个部署在 Vercel 的 Next.js App Router + TypeScript Web 应用，用于物流/快递场景下的批量下单导入。用户上传不同客户提供的 Excel 文件后，系统自动识别模板、完成字段映射、进入类 Excel 预览编辑、实时校验错误、提交下单入库，并提供历史已导入运单查询。

最终提交需要同时提供：

- Vercel 在线访问地址。
- 公司内网 Git 地址，优先使用 `https://git.dev.ztoeco.com/`。

## 2. 已确认事项

1. 数据库使用 Neon PostgreSQL，适配 Vercel 部署。
2. 外部编码为空允许提交，不参与去重。
3. 电话格式采用宽松校验，支持中国手机号和常见座机。
4. “提交下单”在本项目中理解为写入运单数据库，不调用真实物流下单接口。

## 5. 四、功能需求逐条理解

### 5.1 模块一：模板管理与文件导入

需要实现的能力：

- 支持上传 Excel 文件，文件类型包括 `.xlsx` 和 `.xls`。
- 上传方式同时支持拖拽上传和点击选择文件。
- 支持至少 2 种不同模板格式自动识别。本项目目标是兼容题目附件中的 5 种模板。
- 自动识别能力必须覆盖：
  - 不同列名。
  - 不同列序。
  - 表头不在第一行。
  - 合并单元格。
  - 多 Sheet。
  - 缺少选填字段。
- 当自动识别失败或用户需要调整时，用户可以手动选择 Excel 列与系统字段的映射关系。
- 模板记忆学习必须实现：
  - 用户手动调整完列映射关系后，系统自动记录此次映射规则。
  - 记录依据使用表头结构指纹，而不是文件名。
  - 下次上传相同或相似结构模板时，自动应用已保存的映射关系。
  - 用户无需再次手动调整即可继续导入。
- 导入时显示实时进度条：
  - 显示百分比。
  - 显示当前处理条数。
  - 显示总条数。
- 支持 1000 条以上数据导入，页面不能明显卡顿或崩溃。
- 完善错误处理，以下场景需要明确提示：
  - 文件格式错误。
  - 文件为空。
  - 编码异常。
  - Sheet 不存在。
  - 未识别到有效表头。

### 5.2 模块二：数据预览与编辑

需要实现的能力：

- 导入后进入预览列表页面，不能直接提交。
- 预览列表以类 Excel 表格形式展示所有数据。
- 表格要求：
  - 表头固定。
  - 支持横向滚动。
  - 单元格点击可直接编辑。
  - 支持类似在线 Excel 的编辑体验。
  - 支持 `Tab` 和 `Enter` 切换单元格。
- 行内错误实时校验：
  - 必填字段缺失标红。
  - 电话格式错误标红。
  - 重量/件数非正数标红。
  - 温层值不在范围内标红。
- 全部错误一次性展示：
  - 校验时必须同时列出所有错误。
  - 错误信息明确标注行数、字段名和错误原因。
  - 示例：`第 3 行，收件人电话：格式错误`。
  - 不允许一次只提示一个错误，让用户反复修改。
- 外部编码重复检测：
  - 同批次内重复要检测。
  - 与数据库已存在数据重复也要检测。
  - 重复行需要高亮提示，并标注与哪一行重复或与历史运单重复。
- 错误行/错误字段需要有明确文字提示，可以使用 Tooltip 或行内错误文字。
- 支持删除行。
- 支持新增空行。
- 支持将当前预览数据导出为 Excel 文件。
- 导出的 Excel 必须包含用户修改后的最新数据，方便离线修改后重新导入。

### 5.3 模块三：提交下单

需要实现的能力：

- 有错误的行不允许提交。
- 点击“提交下单”后，如果仍有错误，提示用户先修正。
- 提交时显示上传/提交进度条。
- 提交成功后，数据持久化到数据库。
- 返回提交结果汇总：
  - 成功 N 条。
  - 失败 N 条。

### 5.4 模块四：已导入运单列表

需要实现的能力：

- 可查看所有历史已导入的运单记录。
- 历史记录必须从数据库读取，不能只依赖浏览器本地状态。
- 支持按外部编码筛选/搜索。
- 支持按收件人姓名筛选/搜索。
- 支持按提交时间筛选。
- 支持分页展示。

### 6.4 Excel 解析流程

1. 读取工作簿。
2. 自动选择有效 Sheet：
   - 扫描每个 Sheet 的前若干行。
   - 选择可识别字段数最多的 Sheet。
   - 忽略“填写说明”等说明类 Sheet。
3. 自动识别表头行：
   - 扫描前 10 行。
   - 对每行做字段别名匹配。
   - 命中字段最多且达到阈值的行作为表头。
4. 处理合并单元格和二级表头：
   - 优先使用靠近数据区的一行作为字段表头。
   - 上层分组标题仅作为辅助，不作为字段名。
5. 生成表头指纹：
   - 规范化列名。
   - 保留列序。
   - 拼接后生成 hash。
6. 读取模板记忆：
   - 完全匹配指纹则自动应用。
   - 相似结构达到阈值时自动推荐或应用。
7. 使用内置别名词典补齐自动映射。
8. 映射失败或用户需要调整时，展示手动映射控件。
9. 解析数据行：
   - 跳过全空行。
   - 标准化电话、重量、件数、温层。
   - 输出统一订单数组。

### 6.5 校验设计

校验需要返回结构化错误：

```ts
type ValidationError = {
  rowId: string;
  rowIndex: number;
  field: OrderField;
  fieldLabel: string;
  message: string;
};
```

校验规则：

| 规则 | 提示 |
| --- | --- |
| 必填字段为空 | `第 N 行，字段名：不能为空` |
| 电话格式错误 | `第 N 行，字段名：格式错误` |
| 重量非正数 | `第 N 行，重量：必须为正数` |
| 件数非正整数 | `第 N 行，件数：必须为正整数` |
| 温层不合法 | `第 N 行，温层：只能是常温/冷藏/冷冻` |
| 批内外部编码重复 | `第 N 行，外部编码：与第 M 行重复` |
| 历史外部编码重复 | `第 N 行，外部编码：与历史运单重复` |

### 6.6 提交设计

1. 提交前重新运行本地校验。
2. 调用接口检查历史外部编码重复。
3. 如有错误，阻止提交并展示全部错误。
4. 无错误时批量提交至 `/api/orders/import`。
5. 后端创建导入批次。
6. 后端写入运单数据。
7. 返回成功/失败汇总。
